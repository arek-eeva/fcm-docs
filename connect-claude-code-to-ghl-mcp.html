A Field Guide

Connecting Claude Code to your GoHighLevel account.

A step-by-step walk-through for wiring Claude Code to the official GoHighLevel MCP server, so you can manage contacts, conversations, opportunities, calendars and more in plain English from the terminal.

01
The Setup

What you're connecting, and why.

GoHighLevel ships an official MCP (Model Context Protocol) server endpoint. Once it's wired up, Claude Code can read and write GHL contacts, conversations, opportunities, calendar events, custom fields and more, directly from a normal terminal conversation.

The connection is HTTP-based. No package install, no local server to keep running. You give Claude a token, Claude talks to GHL over HTTPS, and that's the whole architecture.

Before you start

You'll need: a GoHighLevel sub-account you own or have admin access to, Claude Code installed and working on your machine, and roughly ten minutes.

02
The Credentials

Generate a Private Integration Token.

Step 01

Open Private Integrations inside your sub-account.

Log into GoHighLevel and switch into the sub-account you want Claude to control. Then go to Settings, Private Integrations. On older agency panels you may see this under Settings, Integrations, Private Integrations.

Step 02

Create a new integration.

Click Create New Integration and give it a clear name such as Claude Code. You don't need a webhook URL.

Step 03

Tick the scopes you actually want Claude to use.

Fewer scopes is safer. A solid starter set covers the everyday workflows without giving away the keys:

  • View Contacts, Edit Contacts
  • View Conversations, Edit Conversations, View and Edit Conversation Messages
  • View Opportunities, Edit Opportunities
  • View Calendars, View Calendar Events, Edit Calendar Events
  • View Custom Fields, View Forms, View Locations

You can always come back and add more scopes later.

Step 04

Copy the token immediately.

Click Create. GoHighLevel shows the token once. Copy it straight into a password manager. If you lose it you'll need to generate a new one.

03
The Address

Find your location ID.

The location ID tells the MCP server which sub-account it should act on. Easiest way to find it:

  • While inside the sub-account, look at the URL. It will contain a path like /v2/location/<LOCATION_ID>/... and that ID is the value you need.
  • Or open Settings, Business Profile and scroll to the bottom. Newer UIs display it there.

One PIT plus one location ID equals one sub-account. To control a different sub-account, generate a separate token under that location.

04
The Wiring

Add the MCP server to Claude Code.

Option A. The one-line command (recommended).

From any terminal, run:

claude mcp add --transport http gohighlevel \
  https://services.leadconnectorhq.com/mcp/ \
  --header "Authorization: Bearer YOUR_PIT_TOKEN" \
  --header "locationId: YOUR_LOCATION_ID"

Add --scope user to make it available across every project on your machine, or leave it off to scope it to the current project folder.

Option B. Edit the config file directly.

If you'd rather see the config, create .mcp.json in your project root, or edit ~/.claude.json for global scope. Either way, add this to the mcpServers block:

{
  "mcpServers": {
    "gohighlevel": {
      "type": "http",
      "url": "https://services.leadconnectorhq.com/mcp/",
      "headers": {
        "Authorization": "Bearer YOUR_PIT_TOKEN",
        "locationId": "YOUR_LOCATION_ID"
      }
    }
  }
}
Better Practice

Keep the raw token out of the config file. Use an environment variable instead:

"Authorization": "Bearer ${GHL_BEARER_TOKEN}"

Then export the variable from your shell profile, export GHL_BEARER_TOKEN=pit-..., or load it from a .env file. That way the config can be committed to git without leaking credentials.

05
The Test

Confirm the connection works.

  1. Quit Claude Code and reopen it (or type /mcp to refresh).
  2. Run /mcp inside Claude Code. You should see gohighlevel listed as connected.
  3. Try a simple read: "List the five most recent contacts in GHL."

If something doesn't work, the failure is almost always one of three things: a mistyped token, the wrong location ID, or a scope you forgot to tick. Double-check those before going deeper.

06
The Guardrails

Treat the token like a password.

  • Never commit the PIT to a git repository. If .mcp.json lives in a tracked project, add it to .gitignore or use the env-var pattern shown above.
  • Tokens are sub-account scoped. To control a different sub-account, create a fresh PIT in that location.
  • Revoke any token at any time from Settings, Private Integrations. Treat that page as your kill switch.
  • The scopes you tick define what Claude can do. If you only ticked View scopes, no write operation can succeed, even if you ask for one.
